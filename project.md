# Autopool + Personal Network Architecture & Flow

## 1. Development Scope & Constraints
- **Restricted Areas**: Do not modify files in the root directory.
- **Active Workspaces**: All development is strictly confined to two directories:
  1. `/AdminPanel`
  2. `/UserPanel`

## 2. Core Business Workflows

### 2.1 The Global Root & Company Wallet
- **Alphanumeric IDs**: User IDs in the system will not be standard auto-incrementing numbers. They will be formatted as **"SA" followed by 6 random numbers** (e.g., `SA521489`).
- **User ID 1 (Global Root - "SAPG")**: The very first user in the matrix is owned by the Company. Its username will be **SAPG**, and its specific, hardcoded ID will always be **`SA000001`**. Whenever the database is created, seeded, or restored, this user must be automatically inserted as the absolute beginning point.
- **Automated Sweeping**: The `SA000001` root account does *not* behave like a normal user regarding withdrawals. Any and all income earned by it (Autopool, Sponsor, Level) is automatically swept directly into the **Company Wallet**.
- **Company Wallet**: A secure, separate ledger that holds system fees (transfer fees, withdrawal fees), package revenue cuts (e.g., the $10 from the $60 package), and all matrix earnings generated by User 1.

### 2.2 User Registration Flow
Because the root directory is off-limits, the registration gateway will live inside the UserPanel directory (e.g., `SITE_URL/UserPanel/register.php?ref=SA521489`). The system operates on a strict closed-loop referral model.
- **Link Requirement**: If a user visits the registration page without a `?ref=` parameter, the form is disabled and displays a message: "Direct registrations are not allowed."
- **Form Fields**: 
  - **Name**, **Email Address**, **Phone Number**, **Password**.
  - **Sponsor Referral ID**: Auto-filled from the URL.
  - **Sponsor Name Lookup**: The form must dynamically display the actual Name of the Sponsor right below the ID field (e.g., "Sponsor Name: JOHN DOE") to confirm accuracy.
- **Internal Registration**: An existing sponsor can also log into their `/UserPanel` and directly register a new user under their network using the same form logic.
- **Post-Registration Success Screen**:
  - The system generates the random alphanumeric ID (e.g., `SA123456`).
  - **Screen Display**: The exact Generated USERID and the Password must be explicitly displayed on the success screen.
  - **Buttons Needed**: "Copy ID", "Share on SMS", and "Share on WhatsApp".
  - **Email Notification**: The system instantly sends a welcome email containing the Username and Password, utilizing SMTP settings from the `.env` file and HTML mail templates stored in `/libs`.

### 2.3 Funding & Wallet System (USDT)
The platform utilizes an internal wallet that must be funded manually via cryptocurrency (USDT).
1. **Deposit Submission Flow (`/UserPanel`)**: 
   - User navigates to the "Funds" section.
   - **Amount Selection**: User can click quick-fill buttons (`$11`, `$30`, `$50`, `$100`, `$200`, `$500`) or manually type an amount.
   - **Payment Info**: The screen displays the Company's USDT Wallet Address and QR Code *(which the Admin can change dynamically in their settings panel)*.
   - **Proof Upload**: The user makes the crypto transfer externally, then enters the TxHash and uploads a screenshot/picture of the proof.
   - The request status is set to: **"Waiting for approval"**.
2. **Admin Verification & Transaction Effects (`/AdminPanel`)**:
   - The Admin sees the request in the "Fund Requests" queue and can Approve or Disapprove it.
   - **If Approved**, the system triggers the following synchronized ledger updates:
     - **Company Ledger**: Increases Total Funds. Narration: *"Fund approved for user [USER_ID]"*.
     - **Unutilized Funds Metric**: Increases.
     - **User's Main Deposit Wallet**: Instantly credited with the amount. Narration: *"Fund request approved"*.

### 2.3 Segregated Earnings Wallets (Crucial Architecture)
Because withdrawal eligibility is tied to the package level, **a user does not have a single "Earnings Wallet"**. Instead, earnings are segregated by package level.
- A user will have 6 distinct earning wallets (e.g., `$11_wallet`, `$30_wallet`, etc.).
- When the user earns Autopool, Sponsor, or Level income from the $60 package matrix, those funds go strictly into their `$60_wallet`.
- A user can only withdraw from a specific level's wallet if they meet the exact withdrawal criteria (direct downlines) for that specific level.

### 2.3 Package System & Autopools
The system features 6 distinct packages.
- **Package List & Payouts**:

     - **Autopool Income**: $13 (Matrix rules TBD).

     - **Company Revenue**: $5.
  3. **$60 Package**: 
     - **Eligibility**: Must have the $30 package active AND have at least **4 direct downlines** (2 additional).
     - **Autopool Income**: $26 (Matrix rules TBD).
     - **Sponsor Income**: $20. *(Held/pending if sponsor has not purchased the $60 package).*
     - **Level Income**: $4 Total ($1.00 per level for 4 levels).
     - **Company Revenue**: $10.
  4. **$120 Package**: 
     - **Eligibility**: Must have the $60 package active AND have at least **6 direct downlines** (2 additional).
     - **Autopool Income**: $52 (Matrix rules TBD).
     - **Sponsor Income**: $40. *(Held/pending if sponsor has not purchased the $120 package).*
     - **Level Income**: $8 Total ($2.00 per level for 4 levels).
     - **Company Revenue**: $20.
  5. **$240 Package**: 
     - **Eligibility**: Must have the $120 package active AND have at least **8 direct downlines** (2 additional from the previous 6).
     - **Autopool Income**: $104 (Matrix rules TBD).

     - **Company Revenue**: $80.

- **Global Eligibility Rule (Downline Packages)**: Whenever a package requires a certain number of "direct downlines" to unlock, those downlines **only need to have the base $11 package**. They do *not* need to be upgraded to the higher packages. This rule applies across all package levels.

### 2.4 Infinity Booster System
Alongside the standard packages, the system features a secondary sequential track called the **Infinity Booster**. 
- **Matrix Structure**: Unlike the main packages, the Infinity Booster operates on a **finite 4x2 Matrix structure**. This means each booster board holds exactly **20 downlines total** under a user:
  - **Level 1**: 4 nodes
  - **Level 2**: 16 nodes
- **Booster Levels & Payouts**:
  1. **$10 Booster**: Generates **$50 total** (4x$2.5 + 16x$2.5).
     - **User Earnings**: $20 
     - **Sponsor Income**: $10 *(Held if sponsor not active)*
     - **Upgrade Reserve**: $20 

  2. **$20 Booster**: Generates **$100 total** (4x$5 + 16x$5).
     - **User Earnings**: $40
     - **Sponsor Income**: $20 *(Held if sponsor not active)*
     - **Upgrade Reserve**: $40

  3. **$40 Booster**: Generates **$200 total** (4x$10 + 16x$10).
     - **User Earnings**: $80
     - **Sponsor Income**: $40 *(Held if sponsor not active)*
     - **Upgrade Reserve**: $80

  4. **$80 Booster**: Generates **$400 total** (4x$20 + 16x$20).
     - **User Earnings**: $160
     - **Sponsor Income**: $80 *(Held if sponsor not active)*
     - **Upgrade Reserve**: $160

  5. **$160 Booster**: Generates **$800 total** (4x$40 + 16x$40).
     - **User Earnings**: $320
     - **Sponsor Income**: $160 *(Held if sponsor not active)*
     - **Upgrade Reserve**: $320

  6. **$320 Booster (Final Tier)**: Generates **$1600 total** (4x$80 + 16x$80).
     - **User Earnings**: $1040 (Available for withdrawal).
     - **Sponsor Income**: $160 *(Held if sponsor not active)*
     - **Re-entry IDs**: $400 used to automatically generate **40 active IDs** in the base $10 Booster matrix (creating massive spillover)



### 2.9 Reward Logic Details:
  The reward system is based strictly on a user's *Sponsor Tree* (not the autopool matrix).
  - Every direct referral forms a new "leg".
  - The system continuously calculates the size of the **Strong Leg** (the single leg with the most downlines) versus the sum of **Other Legs**.
  - When specific cumulative targets are hit (e.g., 15 in Strong Leg + 15 in Other Legs), the user is paid a lump sum reward from the reserve pool.

- **Sequential Eligibility**: Packages must be purchased in strict sequential order (1 → 2 → 3 → 4 → 5 → 6). A user cannot skip a level (e.g., must buy $11 before qualifying to buy $30).
- **Simultaneous Operation (No Replacements)**: Upgrading does *not* replace the previous package. If a user buys the $30 package, both their $11 package and $30 package run simultaneously and independently.
- **Independent Autopools**: Each of the 6 packages operates its own entirely separate 2xN autopool matrix. The $30 matrix consists *only* of $30 packages. Buying Package 2 enters the user into the Package 2 matrix independently of their position in the Package 1 matrix.

### 2.7 Package & Booster Purchase Flow
Once a user has funds in their Main Wallet, they can purchase standard packages or Infinity Boosters. Both follow the exact same flow:
- **Purchase Targets**: A user can buy a package/booster for themselves, or enter another user's ID to buy it on their behalf using their own wallet funds.
- **Smart UI (Eligibility Display)**: 
  - The page must **only display the next eligible package/booster** the user is allowed to purchase. 
  - If the user does not meet the eligibility criteria (e.g., missing direct downlines), the UI must explicitly show them **exactly what is missing**.
- **Transaction Effects & Ledger Narrations**: Upon clicking "Purchase", the following synchronized updates occur:
  - **User Wallet**: The package cost is deducted from the Main Wallet. *(Narration: "Purchased Package [Level] for User [ID]")*
  - **Company Ledger**: The "Unutilized Funds" metric decreases. The "Invested/Active Funds" metric increases. *(Narration: "User [ID] activated Package [Level]")*
  - **Package Activation**: The user is marked as active for that package level in the database.
  - **Network Entry**: The user is officially placed into the matrix for that specific package, triggering the Autopool, Sponsor, and Level payout algorithms.

### 2.8 Network Exploration (Tree View)
To prevent massive recursive database loads, the system will not attempt to render a complete graphical tree.
- **Tabular Interface**: The user's network is displayed as a data table. 
- **Lazy Loading (One Level at a Time)**: When a user visits their network page, it only queries and displays their immediate downlines for that level. Each member in the table has an **"Explore"** button; clicking it fetches that specific member's immediate downline.
- **Distinct Network Views**: Because the structural logic differs, the user will have three entirely separate tabs/pages to explore:
  1. **Main Autopool Networks** (2xN Matrix - separate view for $11, $30, etc.)
  2. **Infinity Booster Networks** (4x2 Matrix - separate view for $10, $20, etc.)
  3. **Reward / Sponsor Network** (Unlimited width direct referral tree used for Sponsor & Level income rules).

### 2.6 Withdrawals & Internal Transfers
Users can request to cash out their available earnings, or transfer them to their Main Wallet for purchasing.
- **Dynamic Withdrawal Eligibility**: To request an external withdrawal, a user must have **2 more direct downlines** than the requirement of their *highest active package*. 
  - *Example*: If their highest active package is the $120 package (which required 6 direct downlines to purchase), their withdrawal qualification becomes **6 + 2 = 8 direct downlines**. 
- **Internal Transfers & Admin Charges**: If a user transfers funds from any segregated *Earnings Wallet* to their *Main Deposit Wallet* (for the purpose of buying packages/boosters), an **Admin Charge** is applied.
  - The admin charge percentage (e.g., 5% or 10%) is dynamic and will be stored directly in the database per wallet type. 
  - This allows the Admin to set different transfer/withdrawal fees for different wallet levels.
- **External Withdrawal Charge**: A strict fee (defined dynamically in the database alongside internal transfer fees) is deducted from every external USDT withdrawal.

## 3. Admin Panel Architecture
The `/AdminPanel` is the central command center for managing the network, users, and financial health.

### 3.1 Core Financial Metrics (Dashboard)
The admin dashboard must display real-time, pre-calculated financial health metrics:
- **Cumulative Funds**: Total USDT that has ever entered the system via approved deposits.
- **Payout Liabilities (Per Network)**: The total amount of funds currently sitting in users' segregated earning wallets (e.g., total liability in $11 wallets, total liability in $30 wallets, plus any "Held" sponsor bonuses).
- **Unused Funds**: The total sum of all USDT sitting idly in users' Main Deposit Wallets (deposited, but not yet spent on packages or withdrawn).
- **Actual USDT Payouts**: The total sum of USDT that has successfully left the system and been sent to external crypto wallets via approved withdrawals.

### 3.2 User Management Module
- **Categories**: All Members, Active Members, Inactive Members, User Details, Blocked Users, Block User Report, Active Member Fund.
- **Admin Impersonation**: Inside the "All Members" list, there must be a **"Login as User"** button allowing the Admin to securely bypass authentication and view a specific user's `/UserPanel` exactly as they see it.

### 3.3 Transactions & Requests
- **Interactive Tree**: A visual network explorer allowing the admin to traverse the entire 2xN main matrices and 4x2 booster matrices.
- **Deposit & Withdrawal Requests**: Dedicated queues for the Admin to review proofs, approve, or reject USDT movements.
- **Transaction Logs**: Detailed ledgers for all internal and USDT transactions.
- **Support Tickets**: A centralized helpdesk to manage user queries.

### 3.4 System Administration
- **Announcements**: A tool to broadcast alerts or news directly to the UserPanels.
- **Global Config (.env File)**: Global site settings (like the Website Name, Logo paths, SMTP credentials, etc.) will be strictly stored in a `.env` configuration file rather than the database for faster loading and secure environment-specific configuration.
- **Dynamic Fees**: Dynamic configurations like the Admin Charge/Transfer fees per wallet type will remain in the database (`wallet_configurations`) so the admin can change them via the dashboard without touching the server environment.

## 4. UI/UX Conventions (Frontend)
- **Strict Popup Rule**: **We will ONLY use fancy CSS/custom popups (e.g., SweetAlert or custom modals). We will NEVER use default JavaScript `alert()` or `confirm()` dialogs anywhere in the application.**

## 5. Database & Performance Requirements (To Be Modeled)
- **Performance Rule (Pre-calculation)**: To prevent expensive `SUM()` and `COUNT()` queries from slowing down the UserPanel dashboard, all accumulated values (e.g., total wallet balances, total network counts, total direct downlines) must be **pre-calculated and stored** directly on the user's record or a dedicated summary table. They will be updated incrementally upon every transaction.
- **Server-Side Pagination**: Every data table (Transactions, Network Tree, User Lists) MUST use strict server-side pagination (e.g., SQL `LIMIT` and `OFFSET`). We will not fetch large data sets for client-side JS caching.
- `deposit_requests`: Tracks user_id, tx_hash, proof_image, amount, status.
- `withdrawals`: Tracks requested_amount, fee_amount, net_amount, destination, status, and **which package wallet** the funds were withdrawn from.
- `wallet_configurations`: A table to store the dynamic **admin charge / fee percentages** for each of the segregated earning wallets.
- `user_wallets`: A new table (or JSON column) to track distinct balances per user: `deposit_balance` (Main Wallet) and segregated `earnings_balance` for every active package/booster. *(These act as the pre-calculated ledgers).*
- `package_matrices`: Since each package and booster has its own autopool, we need a way to track a user's position (`upline_id`, `position`) separately for all matrices.
